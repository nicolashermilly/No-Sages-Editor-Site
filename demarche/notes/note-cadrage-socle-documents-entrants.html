Note de l'éditeur

En cadrant les chantiers un par un — chèque, puis livraison, puis facture — le même besoin revenait à chaque fois : lire un document, le rapprocher, le faire valider par un humain. Ce n'est pas une coïncidence, c'est une architecture. Cette note prend de la hauteur : elle décrit le socle commun qui traite tout document entrant, et montre que nos cinq chantiers n'en sont que des déclinaisons. On construit le socle une fois, on décline le reste.

01

Le constat

Cinq chantiers cadrés séparément partagent le même squelette.

Le chèque à encaisser, le bon de livraison du fournisseur, la facture d'achat, la note de frais, la facture électronique reçue : tous sont des documents qui entrent dans l'entreprise et qu'il faut traiter. En les cadrant un par un, un motif s'est imposé — chacun suit exactement la même séquence : on capture le document, on en extrait les données, on les rapproche à l'existant, un humain valide, une action métier se déclenche, et le document est archivé.

Traiter ces cinq chantiers comme des projets indépendants reviendrait à construire cinq fois le même moteur, avec cinq variantes de bugs. La bonne échelle n'est pas le document : c'est le parcours.

02

Le parcours commun

Six étages, du document brut à l'action comptable. Les mêmes pour tous.

  1. 1

    Capture tronc

    Le document entre : photo, PDF, ou fichier électronique structuré. Import unifié, quelle que soit la source.

  2. 2

    Extraction tronc

    OCR / vision pour les documents image (le cœur qui manque aujourd'hui), ou parseur pour les documents déjà structurés (Factur-X). Sortie : des données normalisées.

  3. 3

    Rapprochement tronc + règles

    Lier le document à l'existant : le BL à sa commande, la facture à ses BL, le chèque à ses factures. Moteur générique, règles par type de document.

  4. 4

    Validation manuelle tronc

    La machine propose, l'humain confirme. Aucune écriture sans validation explicite. Règle absolue, identique partout.

  5. 5

    Action métier produit

    Le seul étage vraiment spécifique : réception + stock (BL), écriture comptable (facture, chèque), remboursement (note de frais). Chaque domaine branche son action.

  6. 6

    Archivage tronc

    Le document est classé en GED, traçable, opposable. Mécanisme unique pour tous.

Cinq étages sur six sont communs. Seul l'étage 5 (l'action métier) change d'un document à l'autre. C'est toute la démonstration : le socle est massivement mutualisable.

03

Les cinq domaines, une seule matrice

Chaque chantier déjà cadré est une instance du même parcours. La preuve en tableau :

ChèqueimageOCR→ facturesRèglement + tiers-payeur
Bon de livraisonimage/PDFOCR→ commandeRéception + entrée stock
Facture d'achatimage/PDFOCR→ BL + commandeÉcriture comptable (achat)

Une seule colonne varie réellement : l'action métier. La note de frais n'a même pas de rapprochement (pas de commande en amont). La facture électronique remplace l'OCR par un parseur XML — mais le reste du parcours est identique. Ce ne sont pas cinq projets. C'est un socle et cinq adaptateurs.

04

Le mur commun : l'extraction

Un obstacle revient dans les cinq chantiers, et il est technique : lire un document image.

Le moteur de génération de contenu (Eliot) est aujourd'hui texte seul : il ne sait pas ingérer une image ou un PDF pour en extraire des données. Or les cinq documents (sauf la facture électronique, déjà structurée) arrivent en image. L'extraction visuelle — transformer une photo de chèque, de BL, de justificatif en données exploitables — est le seul étage qui n'a aucun appui existant.

C'est précisément pourquoi il doit être construit dans le socle, une seule fois : ce serait une erreur de le coder pour le chèque, puis de le refaire pour le BL, puis pour la facture d'achat. Le seam d'extraction visuelle est le cœur transverse à bâtir en premier.

05

Tronc commun : construit une fois, partagé deux fois

Le socle n'est pas propre à un produit : Anim'Gest et Cockpit'Gest le partagent.

Anim'Gest · avance

Terrain d'expérience

Anim'Gest a déjà travaillé l'OCR (chèques, justificatifs) et la facturation électronique. Son existant nourrit la conception du socle.

  • Pipeline OCR déjà amorcé (à constater)
  • Expérience du rapprochement et des statuts
  • Source d'inspiration pour le seam d'extraction

Cockpit'Gest · socle récent

Chaîne aval prête

Cockpit'Gest a déjà l'aval (réception, stock, GED, comptabilité de vente). Il lui manque surtout l'amont : capture et extraction.

  • Réception + stock + GED réutilisables
  • Eliot présent mais texte seul (extraction à ouvrir)
  • Comptabilité versionnée (hors procédure stockée)

Le même mur (extraction visuelle) se dresse dans les deux produits. Le construire une fois, dans le tronc, agnostique, évite de le résoudre deux fois. C'est la même discipline que pour la charte ou les notes de frais : un cœur commun, des adaptateurs par produit.

06

Ce qui est socle, ce qui est adaptateur

Socle tronc (construit une fois)

Le moteur transverse

  • Capture unifiée (image / PDF / structuré)
  • Extraction : seam vision (OCR) + parseur (Factur-X)
  • Moteur de rapprochement générique
  • Cycle de validation manuelle
  • Archivage GED

Adaptateur par document (léger)

La déclinaison

  • Mapping des données extraites vers le document
  • Règles de rapprochement propres au type
  • Action métier (réception, compta, remboursement)
  • Écran dédié

Un nouveau type de document entrant, demain, ne sera plus un chantier : ce sera un adaptateur. Le coût du premier document est élevé (il construit le socle) ; le coût des suivants s'effondre. C'est là tout l'intérêt de la vue macro.

07

Ce que ça change dans l'ordre des travaux

La priorité n'est plus « quel document », mais « quel étage du socle ».

Certains étages du socle existent déjà côté Cockpit (rapprochement possible sur les commandes, GED, chaîne réception/stock, comptabilité de vente). L'étage manquant et structurant, c'est l'extraction visuelle. Deux voies se dégagent :

  • Livrer d'abord sans extraction — le rapprochement et l'action métier fonctionnent avec une saisie manuelle des lignes. Un bon de livraison peut être créé, rapproché, validé et générer une réception sans OCR. L'extraction n'est qu'une accélération de saisie ajoutée ensuite ;
  • Construire le seam d'extraction en parallèle — comme brique tronc, en s'inspirant de l'existant Anim'Gest, sans bloquer les premiers livrables ;
  • Puis brancher l'extraction sur chaque adaptateur — le socle prêt, chaque document gagne l'OCR sans le réinventer.

Autrement dit : on avance le métier tout de suite (rapprochement, validation, action) là où l'aval est déjà prêt, et on bâtit le socle d'extraction en fond, une fois, pour tous. Le concret n'attend pas ; le socle ne se code pas cinq fois.

Cinq documents, un socle, des adaptateurs.

Le chèque, le bon de livraison, la facture d'achat, la note de frais et la facture électronique ne sont pas cinq projets : ce sont cinq visages d'un même parcours — capturer, extraire, rapprocher, valider, agir, archiver. En construisant ce socle une fois, dans le tronc, partagé par Anim'Gest et Cockpit'Gest, chaque nouveau document devient une simple déclinaison. C'est la vue d'ensemble qui transforme une pile de chantiers en une architecture.
